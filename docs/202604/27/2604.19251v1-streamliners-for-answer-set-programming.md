## 摘要
流线型约束（Streamliner constraints）通过排除部分解空间来缩小组合问题的搜索空间。我们将 StreamLLM 方法（该方法利用大语言模型 (LLM) 为约束编程生成流线型约束）适配到答案集编程 (ASP) 中。给定一个 ASP 编码和少量小型训练实例，我们提示多个 LLM 提出候选约束。导致语法错误、使可满足实例变为不可满足、或在所有训练实例上降低性能的候选约束将被舍弃。幸存的流线型约束与原始编码一起进行评估，我们报告了虚拟最佳编码 (VBE) 的结果，该编码针对每个实例从原始编码及其流线型变体中选择最快的一个。在三个 ASP 竞赛基准测试（伙伴单元问题、推箱子、河内塔）中，VBE 相比原始编码实现了高达 4-5 倍的加速。不同的 LLM 产生了语义多样的约束，而不仅仅是语法上的变体，这表明该方法捕捉到了真实的题目结构。

## Abstract
Streamliner constraints reduce the search space of combinatorial problems by ruling out portions of the solution space. We adapt the StreamLLM approach, which uses Large Language Models (LLMs) to generate streamliners for Constraint Programming, to Answer Set Programming (ASP). Given an ASP encoding and a few small training instances, we prompt multiple LLMs to propose candidate constraints. Candidates that cause syntax errors, render satisfiable instances unsatisfiable, or degrade performance on all training instances are discarded. The surviving streamliners are evaluated together with the original encoding, and we report results for a virtual best encoding (VBE) that, for each instance, selects the fastest among the original encoding and its streamlined variants. On three ASP Competition benchmarks (Partner Units Problem, Sokoban, Towers of Hanoi), the VBE achieves speedups of up to 4--5x over the original encoding. Different LLMs produce semantically diverse constraints, not mere syntactic variations, indicating that the approach captures genuine problem structure.

#### 1. 核心问题与研究动机
*   **核心问题**：答案集编程（ASP）求解器的性能高度依赖于问题的逻辑编码方式。即使是逻辑等价的公式，其求解效率也可能存在巨大差异。
*   **研究动机**：
    *   **流线型约束（Streamliners）**是一种有效的优化手段，通过排除部分解空间（如打破对称性或添加隐含约束）来加速求解，但手动设计这些约束需要深厚的领域知识。
    *   现有的自动生成方法（如基于命题逻辑或归纳逻辑编程）往往受限于接地（grounding）瓶颈或需要复杂的语言偏置（language bias）。
    *   本文旨在将 LLM 自动生成流线型约束的方法（StreamLLM）从约束编程（CP）领域扩展到 ASP 领域，探索 LLM 是否能理解 ASP 的非单调语义并生成有效的辅助约束。

#### 2. 方法论
该研究提出了一套全自动化的流水线，核心步骤如下：
*   **输入**：原始 ASP 编码和少量可在几秒内解决的小型可满足训练实例。
*   **LLM 提示与生成**：使用多种前沿 LLM（如 Claude 4.5 Sonnet, GPT-5 Mini, Gemini 3 Pro 等），通过思维链（CoT）提示词要求其生成 5 个能消除冗余或打破对称性的 Clingo 约束。
*   **多级过滤机制**：
    1.  **语法检查**：剔除导致 Clingo 报错的候选约束。
    2.  **一致性检查**：剔除导致原本可满足的训练实例变为不可满足的约束。
    3.  **性能筛选**：剔除在所有训练实例上都导致运行时间增加的约束。
*   **约束选择**：在 30 分钟的训练时间内循环生成，最终根据“虚拟最佳编码（VBE）”准则，选择在训练集上表现最好的 3 个约束组合。
*   **评估**：将选出的约束分别或组合添加到原始编码中，在更大的测试集上运行，并计算 VBE（即针对每个实例选择所有变体中最快的一个）。

#### 3. 实验设计
*   **数据集/场景**：选取了 2011 年 ASP 竞赛中的三个经典组合问题：
    1.  **伙伴单元问题（Partner Units Problem, PUP）**：复杂的配置问题。
    2.  **推箱子（Sokoban）**：经典的规划问题。
    3.  **河内塔（Towers of Hanoi）**：经典的递归移动问题。
*   **Benchmark 与对比方法**：
    *   **Baseline**：原始 ASP 竞赛编码。
    *   **对比项**：单个流线型约束编码、所有流线型约束组合编码、以及 VBE（虚拟最佳编码）。
    *   **评估指标**：PAR2（惩罚平均运行时间，超时按 2 倍限时计算）和仙人掌图（Cactus plots）。

*   **硬件**：计算节点配备 2.40GHz 的 10 核双路 Intel Xeon E5-2640 v4 处理器。
*   **软件**：使用 Clingo 5.8.0 求解器，Python 3.11.5 环境。
*   **LLM 访问**：通过 OpenRouter API 访问 Claude 4.5、GPT-5 Mini、Gemini 3 Pro、Mistral Large 和 Deepseek V3.2。
*   **时长**：每个问题的流线型约束搜索（训练阶段）限制在 **30 分钟**内。

#### 5. 实验数量与充分性
*   **实验规模**：针对每个问题进行了两次独立的完整运行（Run 1 & Run 2），以观察 LLM 生成结果的方差。
*   **样本量**：测试集包含 18 到 46 个不等的大型实例。
*   **充分性评价**：实验设计较为充分。作者不仅展示了性能提升，还对生成的约束进行了语义分析（分类为对称性打破、隐含约束等），并尝试手动优化 LLM 生成的约束（发现手动优化后的接地规模虽小，但实际求解性能反而下降，证明了 LLM 生成约束在启发式搜索上的微妙优势）。

#### 6. 主要结论与发现
*   **显著加速**：在 VBE 分析下，该方法相比原始编码实现了 **4 到 5 倍** 的加速。
*   **互补性**：不同的流线型约束在不同实例上表现各异，VBE 的大幅提升说明 LLM 生成的约束具有很强的互补性。
*   **语义深度**：LLM 不仅仅是进行语法微调，还能生成复杂的辅助谓词（如河内塔中的 `stable` 谓词）和利用聚合函数（aggregates）的约束，捕捉到了深层的问题结构。
*   **训练效率**：仅需极少量的小型实例（几秒钟可解）即可引导 LLM 生成适用于大规模实例的有效约束。

#### 7. 优点
*   **全自动化**：无需人工干预，无需预设语言偏置或领域模板。
*   **跨模型协作**：利用多种 LLM 的多样性，生成的约束在语义上非常丰富。
*   **实用性强**：在 ASP 竞赛级别的复杂编码上依然能获得显著的性能增益。

#### 8. 不足与局限
*   **VBE 的理想化**：实验结果主要基于虚拟最佳编码（VBE），这是一种“先验预测”的上限。在实际应用中，如何针对特定实例自动选择最合适的流线型变体仍是一个挑战。
*   **接地效率风险**：LLM 生成的部分约束可能导致二次方级别的接地规模（Quadratic grounding size），虽然在本文实验中表现良好，但在更大规模数据上可能成为瓶颈。
*   **基准覆盖面**：目前仅在三个竞赛题目上进行了验证，尚需在更多样化的工业级 ASP 应用中测试其泛化能力。
