## 摘要
使用大语言模型（LLMs）自动化优化建模是实现可扩展决策智能的一条极具前景的路径，但现有方法要么依赖于基于闭源 LLM 构建的代理流水线（agentic pipelines），导致推理延迟较高；要么使用昂贵的进程监督（process supervision）微调较小的 LLM，这往往会过度拟合到单一的求解器 API。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种将数学规划求解器视为确定性交互式验证器的执行验证学习框架。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的公式化方法消除了对进程级监督的需求，并通过切换验证环境而非重建特定于求解器的数据集，实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 上针对 Gurobi、OR-Tools 和 COPT 进行的实验表明，EVOM 的表现与经过进程监督的 SFT 相当或更优，支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效的低成本求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

### 2. 方法论：核心思想与技术细节
作者提出了 **EVOM (Execution-Verified Optimization Modeling)** 框架，其核心流程是一个闭环的“生成-执行-反馈-更新”过程：

*   **求解器条件化目标**：模型输入包含“问题描述 $q$”和“目标求解器标识 $s$”。模型输出结构化的 ` <think>推理过程</think> <code>代码</code> `。
*   **沙箱执行环境**：构建了一个标准化的执行套件，支持 Gurobi、OR-Tools 和 COPT。它能捕获代码是否可运行、求解器状态（是否最优、无解或无界）以及最终的目标函数值。
*   **执行验证奖励函数**：
    *   **格式奖励 ($r_{fmt}$)**：通过正则匹配强制模型遵循特定的输出协议，确保代码可被解析。
    *   **结果正确性奖励 ($rans$)**：仅基于最终结果。如果执行成功且得到的目标值与真实答案在误差范围内（或状态一致），则给予正奖励。**不使用任何中间推理链或参考代码作为监督信号。**
*   **策略优化算法**：
    *   使用 **GRPO**（组相对策略优化）进行无评论员（Critic-free）的优化，通过组内奖励归一化计算优势。
    *   引入 **DAPO**（解耦剪裁与动态采样）来缓解长链推理中的熵崩溃问题，并跳过无意义的失败组。

### 6. 主要结论与发现
*   **性能超越 SFT**：仅使用结果监督的 EVOM 在多个基准上达到或超过了使用昂贵过程监督的 SFT 模型。

### 7. 优点：亮点与创新
*   **低成本高效率**：将求解器作为“免费”的验证器，极大地降低了数据获取成本。
*   **解耦建模逻辑与语法**：通过 RL 引导模型关注数学结构的正确性，而非死记硬背 API 字符串，提升了通用性。
*   **冷启动策略**：为冷门求解器提供了“LLM 翻译 + 少量 SFT + RL 增强”的实用路径。

### 8. 不足与局限
*   **深层语义错误**：错误分析显示，虽然 RL 显著减少了语法和变量类型错误，但对于**复杂约束逻辑**（Constraint Error）的改进相对有限，这说明仅靠结果反馈可能难以完全解决极高难度的逻辑对齐。
*   **冷启动依赖**：对于模型完全没见过的求解器，纯 RL 存在冷启动困难，仍需少量合成数据引导。
*   **计算开销**：RL 训练过程中的采样和频繁调用求解器（沙箱执行）比传统的 SFT 消耗更多的计算资源和时间。
