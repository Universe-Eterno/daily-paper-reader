## 摘要
奖励模型（RM）是使大语言模型（LLM）与人类偏好对齐的核心，为基于人类反馈的强化学习（RLHF）和高级解码策略提供动力。尽管先前的大多数工作集中在单步生成上，但现实世界的应用越来越多地采用多阶段 LLM 流水线，而有效的奖励引导在这些场景中仍有待探索。我们通过组合优化的代码生成来研究这一问题，构建了一个将奖励模型整合到公式化（formulation）和求解阶段的流水线。我们发现了一个关键挑战：奖励模型预测与实际流水线执行结果之间存在不一致性。为解决这一问题，我们提出了流水线适配奖励模型（PARM），该模型利用流水线特定数据和直接偏好优化来使奖励与下游反馈保持一致。我们将 PARM 实例化为一个两阶段流水线（公式化 -> 代码生成），并在四个公开优化基准测试上对其进行评估，对比基准模型和采样方法，测量其执行率和求解准确率。在 GSM8K 上进行的补充跨领域实验评估了其迁移能力。结果表明，PARM 持续提升了流水线输出的质量和稳定性，为多阶段 LLM 推理的奖励建模提供了新的见解。

## Abstract
Reward models (RMs) are central to aligning large language models (LLMs) with human preferences, powering RLHF and advanced decoding strategies. While most prior work focuses on single-step generation, real-world applications increasingly adopt multi-stage LLM pipelines, where effective reward guidance remains underexplored. We investigate this through code generation for combinatorial optimization, constructing a pipeline that integrates reward models into both formulation and solution stages. We identify a critical challenge: inconsistency between reward model predictions and actual pipeline execution outcomes. To address this, we propose the Pipeline-Adapted Reward Model (PARM), which leverages pipeline-specific data and direct preference optimization to align rewards with downstream feedback. We instantiate PARM as a two-stage pipeline (formulation -> code generation) and evaluate it on four public optimization benchmarks, measuring execution rate and solving accuracy against baselines and sampling methods. A supplementary cross-domain experiment on GSM8K assesses transferability. Results demonstrate that PARM consistently improves pipeline output quality and stability, providing new insights into reward modeling for multi-stage LLM reasoning.

#### 1. 核心问题与整体含义
随着大语言模型（LLM）应用日益复杂，任务往往被分解为多阶段的流水线（Pipeline），例如“问题建模 -> 代码编写 -> 执行验证”。然而，传统的奖励模型（RM）主要针对单步生成设计，在多阶段任务中存在**奖励与最终结果不一致**的问题：某个中间步骤（如建模）可能获得高分，但却导致下游步骤（如代码生成）失败。本文提出了 **PARM**，旨在通过流水线自适应训练，使奖励模型能够根据最终的执行反馈来评估中间阶段的质量，从而提升复杂推理任务的端到端成功率。

#### 2. 方法论
PARM 的核心思想是**跨阶段信用分配**和**自动化偏好数据构建**。
*   **核心架构**：以组合优化任务为例，构建了一个两阶段流水线：
    1.  **公式生成器 (Formulation Generator)**：将自然语言描述转化为数学公式。
    2.  **代码生成器 (Solution Generator)**：根据公式编写 Python 代码并调用求解器。
*   **流水线自适应训练 (PAT)**：
    *   **标签定义**：不再依赖人工标注，而是利用最终执行结果。如果一个公式（第一阶段输出）能引导下游生成至少一个正确的代码解，则该公式被标记为正样本；否则为负样本。
    *   **训练算法**：采用**直接偏好优化 (DPO)**。针对每个阶段分别训练专门的奖励模型（$RM_F$ 用于公式，$RM_S$ 用于代码），使其学习预测该输出对最终任务成功的贡献度。
*   **推理策略**：在每个阶段采用 Best-of-N 采样，利用训练好的 PARM 评分并选择得分最高的候选者进入下一阶段。
*   **自我修正 (Self-Debugging)**：集成错误反馈循环，当代码执行失败时，将错误日志反馈给模型进行迭代修复。

*   **跨阶段反馈至关重要**：通过最终执行结果训练的 RM 比通用 RM 更能识别出“看似正确但难以求解”的中间建模错误。

*   **解决信用分配问题**：为多阶段任务中“中间步骤如何评价”提供了一个切实可行的数学框架和工程方案。

#### 8. 不足与局限
*   **信用稀疏性**：在超过 2 阶段的长流水线中，早期步骤获得正反馈的概率会指数级下降（即很难随机采样到一个能通向最终成功的路径），这可能导致训练数据极度不平衡。

*   **计算开销**：Best-of-N 采样虽然效果好，但在推理时需要多次生成，增加了计算延迟和成本。
