## 摘要
使用大语言模型（LLM）自动化优化建模是实现可扩展决策智能的一条有前景的路径，但现有方法要么依赖于基于闭源 LLM 构建的、具有高推理延迟的智能体流水线，要么使用昂贵的步骤级监督微调较小的 LLM，而这往往会过拟合到单一的求解器 API。受具有可验证奖励的强化学习启发，我们提出了执行验证优化建模（EVOM），这是一种将数学规划求解器视为确定性交互式验证器的执行验证学习框架。给定自然语言问题和目标求解器，EVOM 生成特定于求解器的代码，在沙箱环境中执行，并将执行结果转换为标量奖励，通过 GRPO 和 DAPO 在“生成-执行-反馈-更新”的闭环过程中进行优化。这种仅基于结果的建模方式消除了对步骤级监督的需求，并通过切换验证环境（而非重新构建特定于求解器的数据集）实现了跨求解器的泛化。在 NL4OPT、MAMO、IndustryOR 和 OptiBench 数据集上，针对 Gurobi、OR-Tools 和 COPT 的实验表明，EVOM 的表现达到或超过了经过步骤监督的有监督微调（SFT），支持零样本求解器迁移，并通过在目标求解器后端下持续训练实现了有效且低成本的求解器适配。

## Abstract
Automating optimization modeling with LLMs is a promising path toward scalable decision intelligence, but existing approaches either rely on agentic pipelines built on closed-source LLMs with high inference latency, or fine-tune smaller LLMs using costly process supervision that often overfits to a single solver API. Inspired by reinforcement learning with verifiable rewards, we propose Execution-Verified Optimization Modeling (EVOM), an execution-verified learning framework that treats a mathematical programming solver as a deterministic, interactive verifier. Given a natural-language problem and a target solver, EVOM generates solver-specific code, executes it in a sandboxed harness, and converts execution outcomes into scalar rewards, optimized with GRPO and DAPO in a closed-loop generate-execute-feedback-update process. This outcome-only formulation removes the need for process-level supervision, and enables cross-solver generalization by switching the verification environment rather than reconstructing solver-specific datasets. Experiments on NL4OPT, MAMO, IndustryOR, and OptiBench across Gurobi, OR-Tools, and COPT show that EVOM matches or outperforms process-supervised SFT, supports zero-shot solver transfer, and achieves effective low-cost solver adaptation by continuing training under the target solver backend.

### 1. 核心问题与研究动机
*   **研究背景**：将自然语言描述的实际问题转化为数学优化模型（如线性规划 LP、混合整数规划 MILP）是运筹学（OR）的核心，但极具挑战性，通常需要专家参与。
*   **核心问题**：现有的 LLM 自动化建模方法存在两大瓶颈：
    1.  **推理成本与延迟**：基于 Prompting 的智能体系统（Agentic pipelines）多依赖闭源大模型，推理轮次多、延迟高。
    2.  **监督成本与泛化性差**：基于微调（SFT）的方法依赖昂贵的“过程监督”（即需要标注中间推理步骤和参考代码），且模型容易过拟合到特定求解器的 API 语法，导致在不同求解器间迁移困难。
*   **研究动机**：受可验证奖励强化学习（RLVR）启发，探索是否能利用数学规划求解器作为“天然验证器”，仅通过执行结果反馈来训练模型，从而降低数据成本并提升跨求解器的泛化能力。

### 2. 方法论：EVOM 框架
*   **核心思想**：提出 **EVOM（Execution-Verified Optimization Modeling）** 框架，将优化建模视为一个“生成-执行-反馈-更新”的闭环过程。
*   **关键技术细节**：
    *   **求解器即验证器**：将 Gurobi、OR-Tools 等求解器封装在沙箱环境中，作为确定性的反馈来源。
    *   **输出协议**：强制模型输出结构化的 `<think>`（推理过程）和 `<code>`（求解代码）块。
    *   **奖励函数设计**：
        *   **格式奖励 ($r_{fmt}$)**：检查输出是否符合特定的 XML 标签格式。
        *   **结果奖励 ($r_{ans}$)**：根据代码是否可执行、求解状态是否正常（如 OPTIMAL）、以及最终目标值（Objective Value）是否与真值一致来给出标量奖励。
    *   **算法流程**：
        1.  **生成**：针对给定问题和目标求解器标识，采样一组候选响应。
        2.  **执行**：在沙箱中运行代码，捕获目标值或错误日志。
        3.  **反馈**：计算组内相对优势（Group-relative advantages）。
        4.  **更新**：使用 **GRPO**（组相对策略优化）或 **DAPO** 算法更新策略模型，无需额外的 Critic 模型。

### 6. 主要结论与发现
*   **结果监督的有效性**：EVOM 在不使用任何中间步骤标注的情况下，表现达到或超过了经过昂贵过程监督微调的 ORLM 模型。
*   **强大的迁移能力**：EVOM 训练的模型在跨求解器任务中表现稳健，而 SFT 模型在切换求解器后往往因语法过拟合而性能崩溃。
*   **推理与代码的解耦**：强化学习不仅优化了代码生成的正确性，还隐式地提升了模型的数学推理逻辑。
*   **低成本适配**：通过“环境切换”，模型可以快速适应新求解器的 API 和错误模式，无需重新构建大规模数据集。

### 7. 优点（亮点）
*   **数据效率高**：仅需“问题-答案”对即可训练，大幅降低了数据准备门槛。
*   **泛化性强**：通过将求解器视为环境的一部分，使模型学习到更本质的数学建模逻辑而非死记硬背 API。
*   **闭环自进化**：利用确定性反馈进行策略对齐，比依赖人类偏好的 RLHF 更客观、更稳定。

### 8. 不足与局限
*   **语义错误瓶颈**：细粒度错误分析显示，虽然 RL 显著减少了语法和变量类型错误，但对于极其复杂的约束逻辑（语义层面）的提升仍有天花板。
*   **冷启动挑战**：对于 LLM 完全陌生的求解器，纯 RL 探索效率低，仍需极少量（如 100 条）的翻译数据进行初始化。
*   **搜索空间限制**：目前主要针对数学规划问题，对于更广泛的启发式算法或组合优化问题的适用性尚待验证。
